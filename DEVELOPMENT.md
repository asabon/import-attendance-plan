# 開発ガイドライン

本プロジェクトの開発手順とコーディングルールについて記載します。開発に参加するメンバー（人間の開発者および AI アシスタント）は、必ず以下のルールに従ってください。

---

## 1. 開発フロー
本プロジェクトでは、`main` ブランチへの直接 push は禁止されています。必ず以下の手順に従って開発を進めてください。

1. GitHub Issue の発行
2. 新しい作業ブランチの作成（命名規則: `<Issue番号>-<作業概要>` を厳守。作業概要は半角英数字とハイフンのみ）
3. コードおよびテストの修正・追加
4. ローカルでのテスト実行およびパスの確認
5. プッシュおよびプルリクエスト（PR）の発行
6. レビュー後、マージおよびブランチの削除

---

## 2. コーディングルール：JSDoc とテストコードの同期

コードの品質と可読性を保つため、**本番コード（.gs）の JSDoc コメント** と **テストコード（.test.js）** の内容を常に同期させてください。

### ルール
* **テスト仕様のドキュメント化**:
  `.gs` 内の主要な関数には必ず JSDoc コメントを付与し、その中に `[テストされている仕様]` セクションを設けて、テスト対象の挙動を日本語で箇条書きしてください。
* **一対一の同期**:
  * **テストを追加・変更した場合**: 対応する `.gs` ファイル側の JSDoc 内の `[テストされている仕様]` も漏れなく更新してください。
  * **本番コードを修正・追加した場合**: 同様に JSDoc 内の仕様を更新し、対応するテストコード側にもそれを検証するテストケースを追加してください。
  * JSDoc に書かれている仕様と、テストコードで実際に `expect` している内容に乖離があってはなりません。

### JSDocの記述例 (`Parser.gs` より)
```javascript
/**
 * メール本文をパースし、登録・削除の操作配列を返します。
 * 同一日付に対する操作が複数ある場合は最後の操作を優先します。
 * 
 * [テストされている仕様]:
 * - `set` (登録) と `delete` (削除) コマンドが正しくパースされること
 * - 同一日付に対する指示が複数ある場合、最後の指示が優先されること（重複排除）
 * - 不正なコマンド（`set`/`delete`以外）がある場合、エラーをスローすること
 * 
 * @param {string} body - メールの本文
 * @returns {Object[]} パースされた操作オブジェクトの配列
 */
function parseAttendanceMailBody_(body) {
  // ...
}
```

## 3. テストの実行方法

本プロジェクトでは **Jest** を使用してローカル環境で Google Apps Script の単体テストを行います。

### ローカルでの実行
以下のコマンドでテストを実行できます。

```bash
# 依存関係のインストール (初回のみ)
npm install

# テストの実行
npm test
```

### CI (GitHub Actions)
GitHub に push または PR が送信された際、自動的に GitHub Actions 上でテストが実行されます。マージ前に必ずテストがパスしている（グリーンになっている）ことを確認してください。

## 4. Git フックの設定 (ローカルでの main ブランチ直接コミット防止)

誤ってローカルの `main` ブランチへ直接コミットしてプッシュ失敗することを防ぐため、コミット時にカレントブランチを判定して `main` への直接コミットをブロックする Git フック（`pre-commit`）を導入しています。

### セットアップ方法
リポジトリをクローンした後、以下のコマンドを実行してリポジトリ内の `.githooks` ディレクトリをフックのパスとして登録してください。

```bash
git config core.hooksPath .githooks
```

これにより、ローカルの `main` ブランチで `git commit` を実行した際に自動でコミットがブロックされるようになります。

## 5. AI アシスタント（Antigravity等）向け特別ルール

AI アシスタントが本リポジトリで作業を行う際は、本ガイドラインのルールに加え、以下のプロセスを厳格に順守してください。

### ① Issue 起票と作業ブランチ作成の義務化
- コードやドキュメントに何らかの修正を加える際は、必ず事前に GitHub Issue を起票してください。
- 起票した Issue 番号を含む作業ブランチ（命名規則: `<Issue番号>-<作業概要>` を厳守。作業概要は半角英数字とハイフンのみ）を新規作成し、そのブランチに切り替えてから修正をコミットしてください。
- `main` ブランチ上での直接コミットは禁止します。

### ② コミット前のユーザー確認
- すでにローカルで何らかの作業ブランチに切り替わっている状態でコミットを行う場合、コミットを実行する前に必ず「このブランチに対象の変更をコミットして良いか」をユーザーにチャット上で確認してください。

### ③ プッシュおよびマージの制限
- AI による `main` ブランチへの直接プッシュ、および `main` ブランチへの直接マージ操作は禁止します。
- 作業完了後は必ず Pull Request (PR) の作成までを行い、マージはユーザー（人間）が承認・実行するものとします。

### ④ Windows環境（PowerShell）での文字化け（非ASCII文字）対策
- Windows環境下で `gh` コマンドなどの外部CLIツールを実行する際、日本語などの非ASCII文字を引数（`--body` や `--title` など）に直接指定すると、端末のエンコーディングの不一致により文字化けが発生します。
- **対策**:
  - 本文（Body）などを指定する際は、直接引数で渡すのではなく、一度 UTF-8 でエンコードした一時ファイル（例: `pr_body.txt`）を介して、ファイル読み込みオプション（`--body-file` や `-F`）を使用してください。
  - タイトル等のインライン文字指定が必要な場合やファイルオプションがない場合は、一時的なNode.jsスクリプトを作成して実行するか、PowerShell側で明示的にUTF-8エンコーディングを設定した上で実行してください。

### ⑤ Pull Request 作成時のルール（タイトル・ラベルの付与）
- Pull Request (PR) を作成する際は、Release-Drafterによるリリースノート生成と自動バージョン解決が正しく機能するように、以下の設定を必ず行ってください。
- **PRタイトル**: リリースノートにそのまま掲載されるため、変更内容が第三者にも一目でわかる簡潔で明瞭なタイトルを設定してください。**また、タイトルは必ず日本語で記載してください（英語表記は不可）。**
- **PRラベル**: 変更内容に合わせて、必ず以下の適切なラベルを1つ以上付与してください。
  - `enhancement` / `feature` / `feat` : 新機能・機能追加（マイナーバージョンアップ）
  - `bug` / `fix` : バグ修正（パッチバージョンアップ）
  - `documentation` / `docs` : ドキュメント更新（パッチバージョンアップ）
  - `chore` / `refactor` / `ci` / `build` : 内部的な修正・リファクタ・環境構築など（パッチバージョンアップ）

## 6. リリース手順

本プロジェクトでの新バージョンのリリース作業は、以下の手順に沿って実施します。

### ① リリースバージョンの決定とコードの更新
1. **次のバージョンの確認**:
   GitHub のリポジトリの **Releases** ページを開き、Release-Drafter によって自動作成された下書き（Draft）リリースノートに提案されている次のバージョン番号（例: `v0.2.0`）を確認します。
2. **バージョン更新用ブランチの作成**:
   `main` ブランチから `<次のバージョン>-bump-version` ブランチ（例: `0.2.0-bump-version`）を作成します。
3. **コード内のバージョン情報の更新**:
   [src/Version.gs](file:///e:/work/import-attendance-plan/src/Version.gs) 内のコメント `Version: X.Y.Z` および変数 `IMPORT_ATTENDANCE_PLAN_VERSION` の値のみを、リリースするバージョン（例: `0.2.0`）に書き換えます。他のファイルへのバージョンコメント追加は不要です。
4. **マージ**:
   変更をコミットしてPRを作成し、テストが通ったら `main` ブランチにマージします。（※このPR自体も Release-Drafter によりリリースノートの下書きに自動追加されます）

### ② GitHub でのリリース公開
1. GitHub の **Releases** ページにアクセスし、該当する下書きリリースの「Edit」ボタンを押します。
2. タグ名とタイトルが、コードに反映したバージョン名（例: `v0.2.0`）と一致していることを確認します。
3. 自動生成されたリリースノートを確認し、必要があれば手動で体裁を整えます。
4. **「Publish release」** ボタンを押して、リリースを正式公開します（これにより自動的に GitHub 上で該当コミットに対して Git タグが打たれます）。

### ③ Google Apps Script (GAS) への反映（本家デプロイ）
※開発者（管理者）が検証用 GAS プロジェクトに最新リリースをデプロイする際の手順です。

1. **最新タグの取得**:
   ```bash
   git checkout main
   git pull
   git fetch --tags
   ```
2. **リリース対象のタグをチェックアウト**:
   ```bash
   git checkout v0.2.0
   ```
3. **自動デプロイコマンドの実行**:
   ```bash
   npm run deploy
   ```
   *(※このスクリプトは、現在のコミットが正しいリリース用の Git タグを指しており、かつ `src/Version.gs` のバージョン定義と一致している場合のみ安全に GAS へのプッシュおよびバージョンデプロイを実行します。)*
4. **後片付け**:
   デプロイ完了後、`main` ブランチに戻ります。
   ```bash
   git checkout main
   ```

*(※手動コピペで GAS に導入・運用したい一般ユーザー向けの手順は [MANUAL_SETUP.md](MANUAL_SETUP.md) に記載されています。)*
